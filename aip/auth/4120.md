# Google Cloud Certificate Config

Google Cloud Certificate Config is a JSON configuration file managed by the
gcloud CLI to support GCP projects such as Enterprise Certificate Proxy and
Workload Credentials.

This AIP describes the general structure and conventions pertaining to this
configuration file.

## Guidance

This configuration file **should** be initialized and updated through the gcloud
CLI, as opposed to edited by hand. Manually editing this file is acceptable for
beta features that have not been fully integrated into gcloud CLI.

### Default Location

By default, the certificate config file will be located inside the gcloud config
directory under the name "certificate_config.json". The default file path
varies depending on the OS platform:

- MacOS: ~/.config/gcloud/certificate_config.json
- Linux: ~/.config/gcloud/certificate_config.json
- Windows: %APPDATA%/gcloud/certificate_config.json

The default file path can be overridden by setting the environment variable
`GOOGLE_API_CERTIFICATE_CONFIG`.

### Config Structure

The auth libraries **must** support the following sections in the configuration
file:

- **"cert_configs"**: In general, this section specifies the location and
     identity of the certificate to be used as part of certificate-based
     access.
- **"libs"**: In general, this section specifies the file path of the libraries
     needed to facilitate certificate-related operations.

Below are use-case specific examples of the certificate configuration file.

#### Enterprise Certificate Proxy ####

The [Enterprise Certificate Proxy][0] project adds support for delegating
certificate and private-key access to OS-native keystores, as part of CBA
workflows.

A valid configuration file for ECP **must** include at least one
keystore-specific configuration in the "cert_configs" section AND a path to
the "ecp" binary in the "libs" section. The configuration file for
ECP **should** also include a path to the "ecp_client" shared library as well
as the "tls_offload" shared library, which are addtional dependencies required
when making API calls via gcloud CLI.

Below is an example of a valid configuration for ECP for GECC cert on MacOS for
use with gcloud CLI:

```json
{
  "cert_configs": {
    "macos_keychain": {
      "issuer": "Google Endpoint Verification"
    },
  },
  "libs": {
   "ecp": "<gcloud-install-location>/google-cloud-sdk/bin/ecp",
   "ecp_client": "<gcloud-install-location>/google-cloud-sdk/platform/enterprise-certificate/libecp.dylib",
   "tls_offload": "<gcloud-install-location>/google-cloud-sdk/platform/enterprise-certificate/libtls_offload.dylib",
  }
}
```

A documentation of all supported fields for ECP can be found at the github
project's README file.

#### Workload Credentials ####

Workload credentails support mTLS for applications running inside Google Cloud.
Below is an example of a certificate config configured for workload credentials
with token binding:

```json
{
  "cert_configs": {
    "workload": {
      "cert_path": "path/to/cert/file",
      "key_path": "path/to/key/file",
      "workload_identity_provider": "...",
      "authenticate_as_identity_type": "gsa/native",
      "service_account_email": "..."
    }
  }
}
```

A documentation of all supported fields for Workload Credentials can be found
in [AIP 4118][1]. 

### Priority

If more than one validate certificate configuration is specified in the
cert_configs section, we **must** use the following order of priority:

1. ECP (i.e. macos_keychain, windows_store, pkcs11)
2. Workload Credentials

The priority is currently determined by the most secure to least secure channel
of certificate access.
